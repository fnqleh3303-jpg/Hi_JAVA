# YumYumCoach 자료구조/알고리즘 기획 및 고찰 보고서

## 1. 프로젝트 개요
YumYumCoach는 식단 분석을 활용한 개인 맞춤형 건강 코칭 콘솔 서비스이다. 본 프로젝트는 식단 및 회원 관리의 CRUD 기능을 기반으로, 문자열 탐색 알고리즘과 적절한 자료구조를 적용해 검색 기능과 조회 성능을 개선하는 것을 목표로 한다.

이번 구현에서는 다음 두 가지를 핵심 목표로 설정했다.

1. 회원, 식단, 음식, 커뮤니티 데이터를 목적에 맞는 자료구조로 관리한다.
2. 문자열 검색 기능에 KMP, Brute Force, Rabin-Karp, Boyer-Moore를 각각 배치해 알고리즘별 적용 사례를 만든다.

---

## 2. 문제 정의
기존 CRUD 중심 구조만으로는 다음 한계가 있었다.

- 회원 로그인 시 아이디 탐색이 빈번하다.
- 식단, 음식, 게시글 검색 기능을 단순 contains에만 의존하면 알고리즘 프로젝트의 목적이 약해진다.
- 사용자가 `고구마`, `바나나` 같은 키워드로 검색할 때 관련 식단/음식/게시글을 빠르게 찾을 수 있어야 한다.

따라서 본 프로젝트는 다음 문제를 해결하도록 설계했다.

- 회원 조회 성능 개선
- 식단 키워드 검색 기능 추가
- 음식 DB 검색 기능 추가
- 커뮤니티 게시글 검색 기능 추가
- 메뉴 입력 처리의 유연성 향상

---

## 3. 사용한 자료구조 정리

### 3.1 회원 정보: `HashMap<String, User>`
- 위치: `UserRepository`
- 키: 사용자 ID
- 이유: 로그인, 회원조회, 중복검사가 모두 사용자 ID 기준으로 이루어진다.
- 기대 성능: 평균 조회 O(1)

#### 장점
- 로그인 시 빠른 단건 조회 가능
- 회원가입 시 중복 검사 간단
- 구현이 간결함

#### 단점
- 입력 순서를 유지하지 않음
- 전체 순회 기반 정렬 기능에는 추가 작업 필요

---

### 3.2 식단 정보: `LinkedHashMap<Integer, MealRecord>`
- 위치: `MealRepository`
- 키: 식단 ID
- 이유: 식단 상세 조회, 수정, 삭제가 모두 식단 ID 기반이고, 전체 조회 시 입력 순서를 유지할 필요가 있다.
- 기대 성능: 평균 조회 O(1), 순서 유지 가능

#### 장점
- 식단 ID 기반 CRUD에 적합
- 전체 목록 출력 시 저장 순서 유지

#### 단점
- 인덱스 기반 접근은 불리함
- 범위 검색에는 별도 구조가 더 적합할 수 있음

---

### 3.3 음식 정보: `LinkedHashMap<Integer, Food>`
- 위치: `FoodRepository`
- 키: 음식 ID
- 이유: 음식 선택은 ID 기반이고, 음식 DB 전체 출력 시 삽입 순서 유지가 유리하다.

---

### 3.4 커뮤니티 게시글: `ArrayList<CommunityPost>`
- 위치: `CommunityRepository`
- 이유: 게시글은 현재 조회와 검색 중심이며, 데이터량이 크지 않다. 순차 저장과 전체 순회가 단순해 `ArrayList`가 적합하다.

#### 장점
- 순차 순회가 쉬움
- 구현 단순

#### 단점
- 게시글 ID 기반 단건 조회가 많아지면 Map이 더 적합해질 수 있음

---

### 3.5 식단 내부 음식 목록: `List<MealItem>`
- 위치: `MealRecord`
- 이유: 한 식단에 여러 음식이 포함되며, 영양성분 합산은 순차 순회가 기본이다.

---

## 4. 사용한 알고리즘 정리

### 4.1 Brute Force
- 적용 위치: 메뉴 명령 해석 (`MenuCommandResolver`)
- 사용 예: 숫자 대신 `회원관리`, `로그인`, `식단관리`, `키워드검색` 같은 텍스트 입력 처리

#### 선정 이유
메뉴 입력 후보는 매우 적고, 문자열 길이도 짧다. 이 상황에서는 복잡한 전처리 알고리즘보다 Brute Force가 가장 단순하고 직관적이다.

#### 시간복잡도
- 최악: O(nm)
- 하지만 실제 메뉴 입력 길이와 후보 수가 매우 작아 부담이 거의 없다.

---

### 4.2 KMP
- 적용 위치: 식단 키워드 검색 (`MealManager.searchMealsByKeyword`)
- 사용 예: `고구마`를 입력하면 `고구마`, `고구마샐러드`가 들어간 식단을 검색

#### 선정 이유
하나의 검색어를 여러 음식명에 반복 적용할 때 부분 문자열 탐색을 안정적으로 수행할 수 있다. 실패 함수(부분 일치 테이블)를 이용해 불필요한 비교를 줄일 수 있어 알고리즘 학습 목적에도 적합하다.

#### 시간복잡도
- 전처리: O(m)
- 탐색: O(n)
- 전체: O(n + m)

---

### 4.3 Rabin-Karp
- 적용 위치: 커뮤니티 게시글 검색 (`CommunityManager.searchPosts`)
- 사용 예: 게시글 제목 + 내용에서 키워드 검색

#### 선정 이유
여러 텍스트에 동일 패턴을 반복 검사하는 경우 해시 기반 비교가 유용하다. 제목과 본문을 합친 문자열에 대해 동일 패턴을 검사하는 구조와 잘 맞는다.

#### 시간복잡도
- 평균: O(n + m)
- 해시 충돌이 많으면 최악 O(nm)

---

### 4.4 Boyer-Moore
- 적용 위치: 음식 DB 검색 (`FoodSearchManager.searchFoods`)
- 사용 예: 음식명 부분 검색

#### 선정 이유
실무적으로는 긴 문자열과 큰 텍스트에서 효율적인 경우가 많고, bad character 규칙을 활용하는 대표 문자열 알고리즘이기 때문에 교육적 가치가 높다. 음식명 검색에 배치해 알고리즘 분산 적용 구조를 만들었다.

#### 시간복잡도
- 평균적으로 빠른 편
- 최악 O(nm)

#### 구현상 관찰
현재 구현은 유니코드 전체(65536 크기) bad character 테이블을 매 검색마다 생성하므로, 짧은 한글 문자열 데이터셋에서는 오히려 오버헤드가 크게 나타났다. 이 점은 개선 포인트다.

---

## 5. 알고리즘/자료구조 배치 이유 요약

| 대상 | 적용 자료구조/알고리즘 | 선정 이유 |
|---|---|---|
| 회원 로그인/조회 | HashMap | ID 기준 단건 조회가 핵심이라 평균 O(1) 접근이 필요 |
| 식단 CRUD | LinkedHashMap | ID 기반 CRUD + 전체 목록의 순서 유지 |
| 식단 검색 | KMP | 동일 패턴을 여러 음식명에 반복 탐색하는 구조에 적합 |
| 음식 DB 검색 | Boyer-Moore | 대표 문자열 탐색 알고리즘 적용 사례 확보, 부분 검색 구현 |
| 커뮤니티 검색 | Rabin-Karp | 여러 게시글의 제목+본문에 동일 패턴을 해시 기반으로 반복 탐색 |
| 메뉴 명령 입력 | Brute Force | 입력 길이와 후보 수가 매우 작아 단순한 방식이 가장 효율적 |

## 6. 성능 비교 결과

아래 결과는 동일 프로젝트 소스에 대해 별도 벤치마크 코드를 실행한 값이다. 측정 환경에 따라 수치는 달라질 수 있으므로 절대값보다 경향성을 보는 용도로 사용한다.

### 6.1 회원 조회: List vs HashMap
실험 조건
- 회원 수: 100,000명
- 목표 사용자: 마지막 사용자 `user99999`
- 반복 조회: 5,000회

결과
- List 순차 탐색: **2335.23 ms**
- HashMap 조회: **0.68 ms**

#### 해석
회원 기능은 문자열 탐색 알고리즘보다 `HashMap` 사용 여부가 훨씬 큰 영향을 주었다. 로그인, 회원 중복 검사, 내 정보 조회가 모두 사용자 ID 기반이기 때문에 `HashMap` 선택이 타당하다.

---

### 6.2 문자열 탐색 알고리즘 비교
실험 조건
- 문자열 말뭉치: 음식명 기반 텍스트 50,000건 규모로 확장
- 검색 패턴: `고구마`

결과
- Brute Force: **10.19 ms**
- KMP: **35.16 ms**
- Rabin-Karp: **21.70 ms**
- Boyer-Moore: **1671.22 ms**

#### 해석
이번 구현과 데이터셋에서는 Brute Force가 가장 빨랐고, KMP와 Rabin-Karp도 충분히 빠르게 동작했다. 반면 Boyer-Moore는 이론적으로 유리할 수 있지만, 현재 구현에서는 매 검색마다 큰 bad character 테이블을 생성해 오버헤드가 커졌다.

즉,
- **이론적 우수성**과
- **현재 구현에서의 실제 성능**은
항상 같지 않다.

이 점을 고찰에 반영하는 것이 중요하다.

---

## 7. 식단 검색 기능 검증
프로젝트에는 초기 시드 데이터로 다음 식단이 포함되어 있다.

- 식단 ID 2: `닭가슴살 샐러드 + 고구마`
- 식단 ID 4: `오트밀 + 바나나 + 그릭요거트`
- 식단 ID 10: `고구마샐러드 + 바나나스무디`

따라서 식단 관리 메뉴에서 KMP 검색을 실행하면 다음과 같이 동작한다.

- `고구마` 검색 → ID 2, ID 10 검색 가능
- `바나나` 검색 → ID 4, ID 10 검색 가능

이 구조를 통해 부분 문자열 검색(`고구마` → `고구마샐러드`)도 확인할 수 있다.

## 8. 한계점

1. Boyer-Moore 구현 최적화 부족
   - 현재는 bad character 테이블을 매 호출마다 새로 생성한다.
   - 검색 패턴 캐싱이 없어서 짧은 문자열 검색에 비효율이 있다.

2. 커뮤니티 저장소 자료구조 단순화
   - 현재는 `ArrayList` 기반이라 게시글 ID 기반 수정/삭제가 많아지면 비효율적일 수 있다.

3. 식단 분석은 규칙 기반 점수화
   - 개인 질환, 연령, 성별, 운동량까지 반영하는 고도화는 아직 미구현이다.

4. 파일 입출력 연동 미완성
   - 현재는 시드 데이터 로딩 방식이라 프로그램 종료 후 영속 저장이 되지 않는다.

---

## 9. 개선 방향

1. Boyer-Moore 패턴 전처리 캐싱 적용
2. 게시글 저장소를 `LinkedHashMap<Integer, CommunityPost>`로 확장
3. 식단 검색에 다중 키워드 검색 추가
4. 음식 DB를 외부 파일(xlsx/json)에서 실제 로딩하도록 확장
5. 식단 분석에 사용자 건강 목표와 주간 통계까지 반영

---

## 10. 결론
본 프로젝트는 CRUD 중심의 콘솔 프로그램에 자료구조와 문자열 탐색 알고리즘을 실제 기능과 연결해 적용했다는 점에서 의미가 있다.

특히,
- 회원 기능에는 `HashMap`을 적용해 조회 성능을 크게 개선했고,
- 식단/음식/커뮤니티 검색에는 서로 다른 문자열 탐색 알고리즘을 배치해 각 알고리즘의 적용 사례를 분리했다.

성능 비교 결과, 회원 조회에서는 `HashMap`의 효과가 매우 컸고, 문자열 탐색에서는 이론적 복잡도만이 아니라 구현 방식과 데이터 특성도 실제 성능에 큰 영향을 준다는 점을 확인했다.

따라서 본 프로젝트의 핵심 고찰은 다음과 같다.

> 자료구조와 알고리즘은 “유명한 것”을 쓰는 것이 아니라, 데이터 특성과 실제 사용 패턴에 맞게 선택해야 한다.
